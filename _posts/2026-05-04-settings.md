# 告别一刀切：玩转 Claude Code 的分层配置体系

作为开发者，你一定经历过这种痛苦：公司 IT 部门要求全局锁定敏感目录，你的个人项目需要一套专属 MCP 服务器，而团队仓库又得共享一套 Lint 规则。如果所有配置都塞进一个文件里，维护起来简直是灾难。

Claude Code 深刻理解了这种多层级配置的需求，设计了一套优雅的**分层配置体系**。今天我们就来拆解它，看看如何让你的 AI 编程助手在不同场景下都能精准运转。

## 关键概念：四大作用域

Claude Code 使用**作用域**系统来决定配置的应用范围和共享方式。理解这四个层级，是你掌控配置的关键：

| 作用域 | 位置 | 影响范围 | 是否共享 |
| :--- | :--- | :--- | :--- |
| **托管** | 服务端、系统注册表或 `managed-settings.json` | 本机所有用户 | 是（IT 部署） |
| **用户** | `~/.claude/` 目录 | 你自己，跨所有项目 | 否 |
| **项目** | 仓库内的 `.claude/` 目录 | 仓库的所有协作者 | 是（提交到 Git） |
| **本地** | `.claude/settings.local.json` | 仅你自己，仅当前仓库 | 否（被 Git 忽略） |

**什么时候用哪个？**
- **托管**：组织级安全策略、合规红线（不可被任何人覆盖）
- **用户**：你的个人偏好、全局 API 密钥、跨项目主题
- **项目**：团队共享的权限、MCP 服务器、Hooks
- **本地**：你的个人项目级覆盖，或测试尚未与团队共享的配置

当不同作用域出现冲突时，**更具体的范围优先**。优先级从高到低依次为：托管 > 命令行参数 > 本地 > 项目 > 用户。例如，项目设置拒绝了某项权限，即便用户设置中允许了，该权限也会被阻断。

## 实操演示：打造你的 settings.json

`settings.json` 是官方推荐的配置机制。你可以通过在 REPL 中运行 `/config` 命令来可视化修改，也可以直接编辑 JSON 文件。

一个典型的配置文件如下：

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  },
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp"
  },
  "companyAnnouncements": [
    "Welcome to Acme Corp! Review our code guidelines at docs.acme.com",
    "Reminder: Code reviews required for all PRs",
    "New security policy in effect"
  ]
}
```

**实用小贴士**：强烈建议保留 `$schema` 这一行！它能在 VS Code 或 Cursor 等编辑器中启用自动补全和校验，避免写错字段名。

### 阻断敏感文件访问

在日常开发中，防止 AI 误读你的密钥至关重要。你可以使用 `permissions.deny` 来屏蔽敏感文件：

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./config/credentials.json)",
      "Read(./build)"
    ]
  }
}
```

匹配到这些模式的文件不仅会在文件搜索中隐藏，如果 AI 尝试读取也会被直接拒绝。

## 进阶技巧：沙盒与插件管控

### 系统级沙盒隔离

如果你对安全性有极致要求，可以开启沙盒模式，将 Bash 命令的文件系统和网络访问彻底隔离：

```json
{
  "sandbox": {
    "enabled": true,
    "autoAllowBashIfSandboxed": true,
    "excludedCommands": ["docker *"],
    "filesystem": {
      "allowWrite": ["/tmp/build", "~/.kube"],
      "denyRead": ["~/.aws/credentials"]
    },
    "network": {
      "allowedDomains": ["github.com", "*.npmjs.org", "registry.yarnpkg.com"],
      "deniedDomains": ["uploads.github.com"],
      "allowUnixSockets": [
        "/var/run/docker.sock"
      ],
      "allowLocalBinding": true
    }
  }
}
```

沙盒配置非常灵活，`allowWrite` 和 `denyRead` 等数组型配置在不同作用域间是**合并与去重**的，而非简单覆盖。这意味着 IT 部门在托管层设定的安全路径，与你本地添加的个性化路径会智能合并。

### 企业级插件市场管控

对于企业管理员，可以通过托管设置中的 `strictKnownMarketplaces` 严格限制插件来源。设置为空数组 `[]` 即可实现完全封锁，防止团队引入未经审查的第三方代码：

```json
{
  "strictKnownMarketplaces": [
    { "source": "github", "repo": "acme-corp/approved-plugins" }
  ]
}
```

### 验证你的配置

不确定当前生效的是哪层配置？在 Claude Code 中运行 `/status`，它会清晰列出所有配置层的来源和状态，包括企业远程托管、本地 plist 文件或是项目级 JSON，帮你快速定位配置冲突。
